# SOP 11: Endpoint Security & Malware Response

**Target Incidents:** General IT: Antivirus alerts, suspected ransomware, strange pop-ups, unauthorized software execution, and Endpoint Detection and Response (EDR) automated isolations.
**Document Owner:** IT Operations / Security Operations Center (SOC)
**Last Updated:** February 2026

---

## Chapter 1: Threat Triage & Verification

### 1.1 Reviewing EDR (Endpoint Detection and Response) Dashboards
Traditional Antivirus (AV) relies on known file signatures, but modern malware easily bypasses signatures using polymorphism. Our enterprise relies on an Endpoint Detection and Response (EDR) platform (e.g., CrowdStrike Falcon, Microsoft Defender for Endpoint, or SentinelOne) which monitors behavioral anomalies and process executions in real-time.



**Initial Triage Steps for L1/L2 Agents:**
1. **Acknowledge the Alert:** When a user reports a "virus pop-up" or an automated ticket is generated by the EDR platform, immediately log into the EDR administrative console.
2. **Locate the Endpoint:** Search the console using the hostname (`cmdb_ci`), the user's IP address, or the username to locate the exact machine generating the alert.
3. **Analyze the Alert Details:** Review the incident timeline (Process Tree). Identify the "Root Cause" process. 
   * *Example:* Did the alert trigger because `OUTLOOK.EXE` spawned `CMD.EXE` which then ran an encoded PowerShell script? This is a classic malicious macro execution path.
4. **Determine the Severity Score:** Most EDRs assign a severity (Low, Medium, High, Critical). Any alert tagged as High or Critical involving credential dumping (e.g., accessing `lsass.exe`), ransomware behaviors (mass file encryption), or lateral movement tools (e.g., Cobalt Strike) requires immediate containment.

### 1.2 Differentiating False Positives from True Threats
A major responsibility of IT Operations is filtering out "noise" so the SOC is not overwhelmed by false positives, while ensuring actual threats do not slip through.

**Identifying a False Positive:**
* **Developer Activity:** Developers frequently compile custom code, run unsigned binaries, or use administrative scripts (e.g., Python scripts mapping drives or modifying registries). If the alert originates from a known developer machine and the process is a known internal script, it is likely a false positive.
* **Approved IT Scripts:** Automated SCCM or Intune software deployments can sometimes trigger heuristic alerts due to sudden, mass file modifications. Verify the file path against known IT tools (e.g., `C:\Windows\ccmcache\`).

**Identifying a True Threat (Red Flags):**

* **Living off the Land (LotL):** The alert shows legitimate native Windows tools (`powershell.exe`, `wmic.exe`, `certutil.exe`, `bitsadmin.exe`) making external network connections to unknown IP addresses or downloading executable payloads.
* **Execution from Temp Directories:** Binaries running from `C:\Users\%username%\AppData\Local\Temp` or `C:\ProgramData` are highly suspicious.
* **Obfuscation:** Command lines that are heavily encoded (e.g., `powershell.exe -enc JABzAD0ATgBlAHcALQBPAGIAagBlAGMAdAA...`) are almost universally malicious.
* **Action:** If the activity cannot be definitively proven as an approved business function, treat it as a **True Threat** and proceed immediately to Chapter 2. Do not wait for user permission.

## Chapter 2: Containment Protocols

### 2.1 Executing Endpoint Network Isolation via EDR Console
If a true threat is suspected, the immediate goal is to prevent the malware from spreading laterally to other workstations or servers across the enterprise network. 



**Execution Steps (Network Containment):**
1. **Do NOT shut down the computer.** Shutting down or rebooting destroys volatile memory (RAM) which the SOC needs for forensic analysis, and it can trigger ransomware to begin encrypting upon the next boot.
2. **Initiate EDR Isolation:**
   * In the EDR console, navigate to the affected endpoint's detail page.
   * Click the **Isolate Host** or **Network Containment** button.
   * *What this does:* The EDR software instantly modifies the local Windows Filtering Platform (WFP) firewall at the kernel level. It cuts off all network traffic (LAN, Wi-Fi, Internet) *except* for the secure telemetry tunnel back to the EDR cloud console. 
3. **Verify Containment:** The EDR dashboard should reflect a "Contained" status. The user will immediately lose access to Teams, email, and the intranet.
4. **Communicate with the User:** Contact the user via phone. Inform them: *"Your machine has exhibited suspicious security activity. We have isolated it from the network to protect enterprise data. Please leave the machine powered on and do not attempt to bypass the block."*

### 2.2 Revoking Active Directory Kerberos Tickets for the User
If the malware has stolen the user's session tokens or credentials (e.g., via Pass-the-Hash or Pass-the-Ticket attacks), isolating the physical endpoint is not enough. The attacker may already be using those stolen credentials from a different, unmanaged device.

**Execution Steps (Identity Containment):**
1. Open **Active Directory Users and Computers (ADUC)**.
2. Locate the compromised user account. Right-click and select **Disable Account** to stop any new authentications.
3. **Revoke Cloud Tokens:** Log into the Microsoft Entra ID (Azure AD) admin center.
   * Locate the user > Click **Revoke Sessions**. This immediately invalidates all active Microsoft 365, Teams, and SharePoint access tokens.
4. **Force Password Reset:** Once the threat is fully eradicated and the SOC gives the all-clear, the user's password must be forcibly reset (refer to SOP 08) before the account is re-enabled.

## Chapter 3: Remediation & Restoration

### 3.1 Running Offline Antivirus Scans
For low-level threats (e.g., Adware, potentially unwanted programs (PUPs), or a simple malicious tracking cookie) where the machine was not fully compromised, a deep offline scan can be used for remediation without destroying the user's data.

**Execution Steps:**
1. If the machine is isolated via EDR, the SOC may provide a temporary "lift" of isolation for the IT agent to remotely connect, or the agent must physically visit the machine.
2. Open **Windows Security** > **Virus & threat protection**.
3. Click **Scan options**.
4. Select **Microsoft Defender Offline scan** and click **Scan now**.
5. The PC will reboot into a secure, minimal Windows Preinstallation Environment (WinPE) outside of the main operating system. This prevents rootkits from hiding themselves during the scan.
6. The scan will take approximately 15 minutes. If it successfully removes the payload, retrieve the scan logs and attach them to the incident ticket.

### 3.2 Invoking the Bare-Metal Re-imaging process for compromised machines
For High or Critical severity threats (Ransomware, Trojans, Keyloggers, or any lateral movement attempts), **the operating system can never be trusted again.** Antivirus scans cannot guarantee that advanced persistent threats haven't left behind hidden backdoors or altered core registry keys. The machine must be entirely wiped.

**Execution Steps (Bare-Metal Re-image):**
1. Obtain approval from the SOC that forensic data gathering is complete and the machine is cleared for destruction.
2. Instruct the user to bring the laptop to the local IT Walk-Up desk, or dispatch a technician to retrieve it. **Do not back up the user's local files.** Any local files (e.g., in the Downloads folder) must be considered compromised. (Enterprise data should already be synced to OneDrive/SharePoint).
3. Connect the machine to the dedicated IT build VLAN.
4. Reboot the machine and press `F12` (or the manufacturer-specific key) to enter the PXE (Preboot Execution Environment) Boot Menu.
5. Select the enterprise SCCM/MDT deployment server.
6. Select the **Format and Partition Disk** task sequence to execute a complete wipe of the SSD.
7. Deploy the standard corporate Windows 11 Enterprise image.
8. Provision a temporary loaner laptop for the user while the re-imaging completes.

## Chapter 4: Security Operations (SOC) Handover

### 4.1 Packaging forensic logs and escalating to Cyber Security
L1 and L2 Service Desk agents are responsible for containment and remediation, but they are not malware analysts. True threats must be formally handed over to the Cyber Security / SOC team for deep forensic analysis, reverse engineering of the payload, and enterprise-wide threat hunting.

**Execution Steps:**
1. Do not close the Incident ticket. Reassign the ticket to the **SOC / Cyber Security** queue.
2. Ensure the ticket contains the following mandatory data points:
   * **Hostname and MAC Address** of the infected machine.
   * **Affected Username**.
   * **Timestamp** of the initial alert.
   * **Containment Actions Taken:** (e.g., "Host isolated via CrowdStrike at 14:05 EST. Azure AD sessions revoked at 14:10 EST.")
3. **Log Collection (If requested by SOC):**
   * Some EDRs require a manual trigger to pull full forensic packages. In the EDR console, execute the "Collect Investigation Package" or "Pull System Logs" command.
   * If physical access is required, export the Windows Event Logs (Security, System, and Application) and Sysmon logs to an encrypted USB drive, and securely transfer them to the SOC's forensic drop-zone.

## Chapter 5: Previously Solved Incidents

### Incident: INC0000551
* **Symptom:** User called stating, "My computer is acting crazy, a black box flashed on my screen and now my fans are spinning really loud." Almost simultaneously, an automated High Severity ticket was generated by the EDR platform for `LT-US-11204`.
* **Triage Performed:** L1 Agent reviewed the EDR dashboard (SOP Step 1.1). The process tree showed that the user opened an email attachment named `Invoice_Q3.pdf.exe`. This file spawned `powershell.exe`, which attempted to execute a base64 encoded command to download an external payload from a Russian IP address.
* **Root Cause:** The user fell victim to a phishing email and executed a malicious payload (likely a dropper for ransomware or a remote access trojan).
* **Resolution:** 1. The L1 Agent immediately clicked "Network Containment" in the EDR console (SOP Step 2.1), cutting the attacker's command-and-control connection. 
  2. The user's AD account was disabled and cloud sessions were revoked (SOP Step 2.2).
  3. The ticket was escalated to the SOC (SOP Step 4.1) who confirmed the payload was a known Emotet variant. 
  4. Once the SOC completed their memory dump, the laptop was confiscated and subjected to a bare-metal re-image via PXE boot (SOP Step 3.2).

### Incident: INC0000803
* **Symptom:** EDR flagged a "Suspicious Script Execution" on a developer's workstation (`WS-DEV-9901`). The alert was categorized as Medium Severity.
* **Triage Performed:** Reviewed the EDR timeline. The alert triggered because `python.exe` was used to run a script named `network_mapper.py` which began aggressively scanning internal IP ranges on port 445 (SMB).
* **Root Cause:** Reached out to the developer assigned to the machine. The developer confirmed they were testing a new internal network inventory tool they were coding for the Infrastructure team. 
* **Resolution:** Since the activity was verified as an approved, legitimate business function by a known developer, this was classified as a False Positive (SOP Step 1.2). The agent added an exclusion rule in the EDR console for the specific hash of that Python script, linked the developer's approval email to the ticket, and closed the incident. No isolation was required.
